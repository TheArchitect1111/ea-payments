# EA Factory Master Build

EA Factory is the operating system for turning client information into approved, EA Chassis-connected websites, portals, experiences, and future deployments.

It is not a website builder. It is not an autonomous AI agent. It is a protocol-driven experience generation platform where EA standards make recommendations and humans approve what moves forward.

## Core Philosophy

- Capture expertise.
- Create standards.
- Automate decisions.
- Maintain human approval.
- Deploy consistently.

Every future website, portal, landing page, image, assessment, proposal, and deployment should be generated from protocols instead of starting from scratch.

## Phase Map

```text
Protocol Center
  -> Repo Library
  -> Project Generator
  -> Skin Factory
  -> Codex Builder
  -> Approval Center
  -> Chassis Deployment
  -> Pulse
```

## Phase 1: Protocol Center

Protocol Center is the first foundation layer and the single source of truth for EA standards.

The initial structured seed lives in:

- `lib/ea-factory.ts`
- `GET /api/ea-factory/protocols`
- `GET /api/ea-factory/protocols?q=skin`

Seeded protocols:

- EA Master Protocol
- EA Brand Protocol
- EA Skin Protocol
- EA Chassis Protocol
- EA Website Protocol
- EA Image Protocol
- EA Sales Protocol
- EA Assessment Protocol
- EA Training Protocol

Each protocol supports:

- Category
- Status
- Active version
- Tags
- Purpose
- Governed areas
- Approval requirement
- Version history

Future protocols reserved in the registry:

- EA Social Protocol
- EA Portal Protocol
- EA Experience Rental Protocol
- EA Recruiting Protocol
- EA Nonprofit Protocol
- EA Church Protocol
- EA Membership Protocol

## Phase 2: Repo Library

Repo Library prevents rebuilding approved patterns.

The foundation connects EA Factory to the existing `EA_REPOSITORY_LIBRARY` and adds intelligence fields:

- Tags
- Compatibility
- Animation level
- Storytelling score
- Mobile score
- Performance score

Endpoint:

- `GET /api/ea-factory/repos`

Repo intelligence should be expanded whenever EA approves a new design system, animation library, storytelling repo, portal template, or client-specific chassis pattern.

## Phase 3: Project Generator

Project Generator converts client information into requirements and strategy. It does not generate code.

Endpoint:

- `POST /api/ea-factory/project-brief`

Required input:

```json
{
  "clientName": "Canadian Prospects",
  "website": "https://canadianprospects.ca",
  "industry": "sports recruiting",
  "mission": "Help student-athletes pursue the next level.",
  "goals": ["increase applications", "support families", "show recruiting proof"],
  "desiredOutcome": "a clear path from application to opportunity",
  "selectedProtocolIds": ["ea-master", "ea-skin", "ea-chassis", "ea-website"]
}
```

Output:

- Story framework
- Transformation narrative
- Website structure
- Portal recommendations
- Module recommendations
- Image requirements
- Repo recommendations
- Build requirements
- Codex build prompt
- Approval gate

## Human Approval Rule

Nothing generated by EA Factory deploys automatically.

Approval must review:

- Story
- Visual direction
- Layout
- Images
- Modules
- Branding
- Functionality

Approved actions:

- Approve
- Reject
- Request revision
- Approve with notes

## Skin And Chassis Rule

The skin controls:

- Story
- Imagery
- Typography
- Layout
- Visual identity
- Emotional experience

The chassis controls:

- Authentication
- Navigation
- Modules
- Search
- Help
- Analytics
- Integrations
- Dashboards

Deployment must never overwrite story, imagery, branding, or messaging. Chassis upgrades should preserve the skin.

## Module Deployment Rule

Every module must:

- Self-register
- Create navigation
- Create dashboard widgets
- Create admin pages
- Create permissions
- Create analytics
- Create search support
- Create help support
- Support mobile
- Support Pulse

## Runtime Launcher Phase 1 (2026)

Autonomous **project create → queue → GenerateWorker → UNDER_REVIEW** foundation inside this repo (extends EACP; no Prisma/Redis greenfield).

Pipeline statuses: `CREATED` → `QUEUED` → `GENERATING` → `UNDER_REVIEW` (later: research/build workers).

| Surface | Path |
|---------|------|
| Launch API | `POST /api/launch` |
| Projects API | `GET /api/projects`, `GET /api/projects/{id}`, restart/cancel |
| Queue cron | `GET /api/cron/factory-queue` |
| Queue health | `GET /api/health/factory-queue` |
| Admin board | `/admin/ea-factory/projects` |
| Domain | `lib/factory-project.ts`, `lib/factory-queue.ts`, `lib/factory-workers.ts` |

ChatGPT Action: `launchProject` in `/api/eacp/openapi`. Field demo remains a separate fast path (`launchFieldDemo`).

Deferred to later phases: research/discovery/planning AI, website/portal builders, R2 ingestion, BullMQ/Postgres.
